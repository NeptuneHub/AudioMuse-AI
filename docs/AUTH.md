# Authentication

Authentication supports three modes via `AUTH_MODE`:

- `local`: built-in AudioMuse login page (`/login`) with JWT session cookie.
- `proxy`: trust reverse-proxy identity headers (for example Authentik in front of AudioMuse).
- `none`: disable authentication checks.

If `AUTH_MODE` is not set, the legacy `AUTH_ENABLED` fallback is still supported:

- `AUTH_ENABLED=true` -> behaves like `AUTH_MODE=local`
- `AUTH_ENABLED=false` -> behaves like `AUTH_MODE=none`

The `API_TOKEN` can be used for machine-to-machine/plugin calls in both `local` and `proxy` mode.
The `JWT_SECRET` is only used by `local` mode (to keep browser sessions stable across restarts).

If you deploy to Kubernetes, **do not place these values in a ConfigMap**; use a Secret resource so the credentials aren’t stored in plaintext imagery or git history:

```yaml
# Local mode example
AUTH_MODE=local
AUDIOMUSE_USER=alice
AUDIOMUSE_PASSWORD=secret123
API_TOKEN=foo-bar-baz
JWT_SECRET=<random-string>
```

```yaml
# Proxy mode example (e.g. Authentik/Caddy/Traefik)
AUTH_MODE=proxy
API_TOKEN=foo-bar-baz
```

In `local` mode, the web UI provides a `/login` page where the user posts the username/password and receives a JWT cookie on success. Subsequent browser requests are authenticated via that cookie.

In `proxy` mode, AudioMuse expects a trusted identity header from your reverse proxy (for example `Remote-User`, `X-Forwarded-User`, `X-Remote-User`, or `X-Authentik-Username`).

Machine‑to‑machine callers may bypass the login page by supplying the `API_TOKEN` in an `Authorization: Bearer …` header. For example:

```bash
curl -v \
  -X POST 'http://192.168.3.233:8000/api/analysis/start' \
  -H 'Authorization: Bearer 123456' \
  -H 'Content-Type: application/json' \
  -d '{}'
```

> NOTE: this warning only applies to `AUTH_MODE=local` if local credentials are not configured.

```
Local authentication is active and one or more credentials were autogenerated and visible in FLASK log. You should set the final values for AUDIOMUSE_USER and AUDIOMUSE_PASSWORD in your environment variables. Optionally set API_TOKEN if you need external/plugin access.
```

## HTTPS

To have a more secure Authentication running everything over HTTPS is needed to avoid that your password go in plain text. This part is something that relay from your infrastracture and not from AudioMuse-AI itself. For example if you're deploy everything on K3S thatr come with Traefik integrated, and you have certmanager with let's encrypt, you can add an IngressRoute like this:

```
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: audiomuse-ingressroute
  namespace: playlist
spec:
  entryPoints:
    - websecure
  routes:
    - match: Host(`playlist.192.168.3.169.nip.io`)
      kind: Rule
      services:
        - name: audiomuse-ai-flask-service
          port: 8000
  tls:
    certResolver: letsencrypt-production
```
